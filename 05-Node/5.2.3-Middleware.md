# 5.2.3 Middleware

Middleware — функции, которые выполняются между получением запроса и отправкой ответа. Через них проходит каждый запрос: логирование, парсинг тела, проверка авторизации, обработка ошибок. Понимание middleware — ключ к тому, чтобы Express-приложение оставалось читаемым при росте.

## 📖 Как работает middleware

Middleware — функция с сигнатурой `(req, res, next)`. Вызов `next()` передаёт управление следующей функции в цепочке. Если `next()` не вызвать, запрос зависнет:

```typescript
import { Request, Response, NextFunction } from "express";

function logger(req: Request, res: Response, next: NextFunction): void {
  console.log(`${req.method} ${req.path}`);
  next(); // передаём управление дальше
}

app.use(logger); // применить ко всем маршрутам
```

Порядок регистрации middleware имеет значение: они выполняются в том порядке, в котором добавлены через `app.use`.

```typescript
app.use(express.json());  // 1. парсим тело
app.use(logger);           // 2. логируем
app.use("/api", router);   // 3. маршруты
```

---

## 📖 Встроенные middleware

**`express.json()`** — парсит тело запроса с `Content-Type: application/json` и кладёт результат в `req.body`.

**`express.urlencoded()`** — парсит тело формы (`application/x-www-form-urlencoded`). Нужен если принимаешь данные из HTML-форм.

**`express.static()`** — отдаёт статические файлы из папки:

```typescript
app.use("/uploads", express.static("uploads"));
// GET /uploads/avatar.png → файл из папки uploads/
```

## 📖 Сторонние middleware: cors и helmet

**`cors`** (Cross-Origin Resource Sharing) — механизм, который контролирует, каким доменам разрешено обращаться к API. По умолчанию браузер блокирует запросы к другому домену, порту или протоколу — это называется политикой одного источника (Same-Origin Policy). Без `cors` фронтенд на `localhost:5173` не сможет достучаться до бэкенда на `localhost:3000`:

```bash
npm install cors
npm install -D @types/cors
```

```typescript
import cors from "cors";

// Разрешить все источники (в разработке):
app.use(cors());

// Разрешить только конкретный источник (в продакшне):
app.use(cors({
  origin: "https://myapp.com",
  methods: ["GET", "POST", "DELETE"],
}));
```

**`helmet`** — выставляет HTTP-заголовки безопасности: запрещает встраивание в iframe, включает защиту от XSS и другие. Одна строка — серьёзная базовая защита:

```bash
npm install helmet
```

```typescript
import helmet from "helmet";

app.use(helmet());
```

Порядок подключения: `helmet` и `cors` — первыми, до роутеров.

## 📖 asyncHandler — избавиться от try/catch

В предыдущем разделе каждый async-обработчик оборачивался в `try/catch`. При 10–15 маршрутах это много повторяющегося кода. `asyncHandler` — обёртка, которая перехватывает ошибку и передаёт её в error middleware через `next(error)`:

```typescript
// src/middleware/asyncHandler.ts
import { Request, Response, NextFunction, RequestHandler } from "express";

export function asyncHandler(
  fn: (req: Request, res: Response, next: NextFunction) => Promise<void>
): RequestHandler {
  return (req, res, next) => {
    fn(req, res, next).catch(next); // ошибка попадает в error middleware
  };
}
```

Использование:

```typescript
import { asyncHandler } from "../middleware/asyncHandler";

router.get(
  "/:city",
  asyncHandler(async (req, res) => {
    const weather = await fetchWeather(req.params.city);
    res.json(weather);
    // try/catch больше не нужен — ошибка уйдёт в error middleware
  })
);
```

## 📖 AppError и error middleware

Когда сервис бросает ошибку, нужно вернуть правильный HTTP-статус. Например, если город не найден — `404`, а не `500`. `AppError` — класс для ошибок с HTTP-статусом:

```typescript
// src/middleware/AppError.ts
export class AppError extends Error {
  constructor(
    public message: string,
    public statusCode: number = 500
  ) {
    super(message);
    this.name = "AppError";
  }
}
```

В сервисе бросают `AppError` с нужным кодом:

```typescript
// src/services/weather.service.ts
import { AppError } from "../middleware/AppError";

export async function fetchWeather(city: string): Promise<WeatherData> {
  const response = await fetch(url);

  if (response.status === 404) {
    throw new AppError(`Город "${city}" не найден`, 404);
  }

  if (!response.ok) {
    throw new AppError("Ошибка внешнего API", 502);
  }

  // ...
}
```

Error middleware — специальная функция с четырьмя параметрами. Express распознаёт её именно по четырём аргументам, поэтому все четыре обязательны:

```typescript
// src/middleware/errorHandler.ts
import { Request, Response, NextFunction } from "express";
import { AppError } from "./AppError";

export function errorHandler(
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction // обязателен, даже если не используется
): void {
  console.error(`[${req.method} ${req.path}]`, err.message);

  if (err instanceof AppError) {
    res.status(err.statusCode).json({ error: err.message });
    return;
  }

  // Неизвестная ошибка — не раскрываем детали клиенту
  res.status(500).json({ error: "Внутренняя ошибка сервера" });
}
```

Error middleware подключают **последним** — после всех роутеров:

```typescript
// src/index.ts
app.use(helmet());
app.use(cors());
app.use(express.json());

app.use("/api/weather", weatherRouter);
app.use("/api/favorites", favoritesRouter);

app.use(errorHandler); // последним
```

## 📖 Middleware для конкретного маршрута

Middleware можно применить не ко всему приложению, а к одному маршруту или роутеру. Это пригодится для авторизации:

```typescript
function requireAuth(req: Request, res: Response, next: NextFunction): void {
  const token = req.headers.authorization;
  if (!token) {
    res.status(401).json({ error: "Требуется авторизация" });
    return;
  }
  next();
}

// Только для этого маршрута
router.get("/profile", requireAuth, asyncHandler(async (req, res) => {
  res.json({ user: "данные профиля" });
}));

// Для всего роутера
router.use(requireAuth);
```

<details>
<summary>💡 Подсказка: requestLogger</summary>

```typescript
import { Request, Response, NextFunction } from "express";

export function requestLogger(
  req: Request,
  res: Response,
  next: NextFunction
): void {
  const start = Date.now();

  res.on("finish", () => {
    const duration = Date.now() - start;
    console.log(`[${req.method} ${req.path}] ${res.statusCode} ${duration}ms`);
  });

  next();
}
```

## 🎯 Финальное задание

Собери все middleware в итоговый `index.ts`:

1. `helmet()` и `cors()` — первыми
2. `express.json()`
3. Кастомный `requestLogger`
4. Роутеры: `/api/weather` и `/api/favorites`
5. Обработчик 404 — если ни один маршрут не совпал:
   ```typescript
   app.use((req, res) => {
     res.status(404).json({ error: `Маршрут ${req.path} не найден` });
   });
   ```
6. `errorHandler` — последним

Проверь: запрос к несуществующему маршруту `/api/unknown` должен возвращать `404`, а запрос к `/api/weather/НесуществующийГород123` — `404` с сообщением из `AppError`.
