# 🟣 5. Node.js

Node.js — это среда выполнения JavaScript на сервере. Здесь строим REST API на Express с TypeScript — бэкенд для приложений.

---

Здесь ты научишься:
- запускать JavaScript вне браузера и понимать, чем серверная среда отличается от клиентской,
- строить REST API с Express и TypeScript,
- разделять код на роуты, контроллеры и сервисы,
- валидировать входные данные и обрабатывать ошибки единообразно,
- реализовывать регистрацию, вход и защищённые маршруты через JWT,
- тестировать API через Vitest и Supertest.

---

> 💡 **Совет:** Весь раздел написан на **TypeScript**. Если ты уже знаком с типами, интерфейсами и async/await — всё это здесь работает так же. Новое — серверная среда выполнения.

---

## 🌤 Сквозной проект: Weather API

Строим бэкенд для погодного приложения.

**Что строим:**

- `GET /api/weather/:city` — прокси к OpenWeatherMap. API-ключ хранится на сервере, браузер его не видит.
- `POST /api/auth/register` и `POST /api/auth/login` — регистрация и вход.
- `GET /api/favorites` и `POST /api/favorites` — избранные города, доступны только авторизованным.
- `POST /api/users/avatar` — загрузка аватарки профиля.

Каждый раздел добавляет новый слой: от первого маршрута до полноценного API с авторизацией и тестами.

### Этап 1: Основы (5.1)

**5.1.1 Что такое Node.js**
- Событийная модель и Event Loop на сервере
- Чем серверная среда отличается от браузера: нет `window`, `document`, `localStorage`
- `process`, `__dirname`, глобальные объекты Node.js
- Встроенные модули: `fs`, `path` — для чего нужны

**5.1.2 Модули и окружение**
- CommonJS (`require`) vs ESM (`import`) — когда что применяется
- `package.json`, `npm`-скрипты, зависимости и devDependencies
- `.env` и `dotenv`: хранение секретов вне кода
- Настройка TypeScript для бэкенда: `tsconfig.json`, `tsx` для запуска

### Этап 2: Express + TypeScript (5.2)

**5.2.1 Express + TypeScript: инициализация проекта**
- Установка `express`, `@types/express`, `tsx`
- Типизация `Request`, `Response`, `NextFunction`
- Первый маршрут: `GET /api/weather/:city` — прокси к OpenWeatherMap

**5.2.2 Роутинг и структура проекта**
- `Router` — разделение маршрутов по файлам
- Структура `routes/` → `controllers/` → `services/`
- Зачем контроллер не должен знать про бизнес-логику, а сервис — про HTTP
- REST-конвенции: методы, статус-коды, именование ресурсов

**5.2.3 Middleware**
- Встроенные: `express.json()`, `express.static()`
- Сторонние: `cors`, `helmet`
- Кастомный middleware: логирование, `asyncHandler`-обёртка
- Error middleware — единая точка обработки ошибок, типизированный `AppError`

**5.2.4 Валидация (Zod)**
- Схемы для тела запроса, параметров и query
- Middleware-валидатор: отклоняет невалидные запросы до контроллера
- Типобезопасные данные в контроллере через `z.infer<>`

**5.2.5 Аутентификация (JWT)**
- Хеширование паролей через `bcrypt`
- Генерация и верификация JWT через `jsonwebtoken` с TypeScript-типами
- `POST /api/auth/register` и `POST /api/auth/login`
- `authMiddleware` — защита маршрутов, `req.user` с типом

**5.2.6 Загрузка файлов (Multer)**
- Multer как middleware: `upload.single()`, `upload.array()`
- Валидация типа файла и максимального размера
- Хранение локально и по публичной ссылке
- `POST /api/users/avatar` — загрузка аватарки

### Этап 3: Тестирование (5.3)

**5.3.1 Тестирование API (Vitest + Supertest)**
- Vitest + Supertest — минимум инструментов, максимум покрытия
- Supertest: HTTP-запросы к Express в тестах без реального сервера
- Unit-тесты для сервисов, интеграционные для маршрутов
- Изоляция через тестовую БД или моки

### Этап 4: Экспертный уровень (5.4)

**5.4.1 WebSockets**
- `socket.io` с TypeScript
- Типизированные события: `socket.emit`, `socket.on`
- Пример: live-обновление погоды при смене города

**5.4.2 Streams и Buffer**
- Что такое Buffer и зачем он нужен
- Readable, Writable, Transform Streams
- `pipe` — цепочки потоков
- Когда Streams нужны, а когда достаточно обычного `await`

**5.4.3 Docker + деплой**
- `Dockerfile` для Node.js-сервиса
- `docker-compose` с бэкендом и MongoDB
- Деплой на Railway или VPS
