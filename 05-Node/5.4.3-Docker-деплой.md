# 5.4.3 Docker + деплой

API работает на твоей машине. Но когда другой разработчик клонирует репозиторий или ты переносишь проект на сервер — что-то идёт не так: другая версия Node.js, не установлены зависимости, нет `.env`. Docker решает это: он упаковывает приложение вместе со всем окружением в образ (image), который запускается одинаково везде.

---

## 🎯 Что ты узнаешь

- Как написать Dockerfile для Node.js-сервиса
- Как уменьшить размер образа через многоэтапную сборку
- Как поднять приложение и базу данных одной командой через docker-compose
- Как управлять переменными окружения в контейнере
- Как задеплоить проект на Railway и на VPS

## 📖 Dockerfile

Dockerfile — инструкция для сборки образа. Каждая строка — слой; Docker кэширует слои и пересобирает только то, что изменилось.

Минимальный Dockerfile для Node.js:

```dockerfile
FROM node:22-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY . .

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

Но у него проблема: TypeScript нужно скомпилировать перед запуском. Если делать это в одном образе, в финальный контейнер попадают devDependencies, исходники `.ts` и кэш компилятора — лишние мегабайты.

---

## 📖 Многоэтапная сборка

Multi-stage build решает это: первый этап (builder) устанавливает все зависимости и компилирует TypeScript, второй этап (production) берёт только результат сборки.

```dockerfile
# ---- Этап 1: сборка ----
FROM node:22-alpine AS builder

WORKDIR /app

# Сначала копируем только package файлы — слой кэшируется,
# пока зависимости не изменились
COPY package*.json ./
RUN npm ci

# Копируем исходники и компилируем
COPY tsconfig.json ./
COPY src/ ./src/
RUN npm run build   # запускает tsc → создаёт папку dist/

# ---- Этап 2: production ----
FROM node:22-alpine AS production

WORKDIR /app

# Берём только production-зависимости
COPY package*.json ./
RUN npm ci --only=production

# Берём скомпилированный код из первого этапа
COPY --from=builder /app/dist ./dist

# Непривилегированный пользователь — хорошая практика безопасности
USER node

EXPOSE 3000
CMD ["node", "dist/index.js"]
```

Финальный образ содержит только `dist/` и `node_modules` с production-зависимостями — без TypeScript-компилятора, без исходников.

Для этого нужно добавить `build`-скрипт в `package.json` и настроить `outDir` в `tsconfig.json`:

```json
// package.json
{
  "scripts": {
    "dev": "tsx watch src/index.ts",
    "build": "tsc",
    "start": "node dist/index.js"
  }
}
```

```json
// tsconfig.json — добавить outDir
{
  "compilerOptions": {
    "outDir": "dist",
    "rootDir": "src"
  }
}
```

## 📖 .dockerignore

`.dockerignore` работает как `.gitignore`, но для Docker — исключает файлы из контекста сборки. Без него `COPY . .` скопирует `node_modules`, `.env` и `dist/` внутрь образа.

```
# .dockerignore
node_modules
dist
.env
.env.*
*.log
.git
coverage
uploads
```

`node_modules` исключают всегда — в контейнере они устанавливаются заново через `npm ci`. `.env` исключают, чтобы не запечь секреты в образ.

## 📖 Установка Docker

Скачай и установи **Docker Desktop** с [docker.com/products/docker-desktop](https://www.docker.com/products/docker-desktop/) — он включает и Docker, и docker-compose. Доступен для macOS, Windows и Linux.

После установки проверь:

```bash
docker --version        # Docker version 27.x.x
docker compose version  # Docker Compose version v2.x.x
```

Если команды не найдены — перезапусти терминал или перезагрузи машину.

## 📖 Основные команды Docker

```bash
# Собрать образ с тегом
docker build -t weather-api .

# Запустить контейнер
docker run -p 3000:3000 weather-api

# Запустить с переменными окружения
docker run -p 3000:3000 \
  -e PORT=3000 \
  -e OPENWEATHER_API_KEY=abc123 \
  -e JWT_SECRET=supersecret \
  weather-api

# Посмотреть запущенные контейнеры
docker ps

# Остановить контейнер
docker stop <container_id>

# Посмотреть логи
docker logs <container_id> -f
```

`-p 3000:3000` — проброс порта: `<порт на хосте>:<порт внутри контейнера>`.

## 📖 docker-compose

Запускать несколько контейнеров вручную неудобно. `docker-compose` описывает все сервисы в одном файле и поднимает их одной командой.

```yaml
# docker-compose.yml
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - PORT=3000
      - OPENWEATHER_API_KEY=${OPENWEATHER_API_KEY}
      - JWT_SECRET=${JWT_SECRET}
      - MONGODB_URI=mongodb://mongo:27017/weatherapp
    depends_on:
      - mongo
    restart: unless-stopped

  mongo:
    image: mongo:7
    volumes:
      - mongo_data:/data/db    # данные переживают перезапуск контейнера
    restart: unless-stopped

volumes:
  mongo_data:
```

`${OPENWEATHER_API_KEY}` — docker-compose читает переменные из `.env` файла в той же папке. Сам `.env` в образ не попадает.

```bash
# Поднять все сервисы в фоне
docker compose up -d

# Посмотреть логи всех сервисов
docker compose logs -f

# Остановить и удалить контейнеры
docker compose down

# Остановить и удалить данные (включая volume с MongoDB)
docker compose down -v
```

`mongo:27017` в `MONGODB_URI` — это имя сервиса `mongo` из compose-файла. Docker автоматически создаёт внутреннюю сеть, где сервисы обращаются друг к другу по имени.

## 📖 Переменные окружения в продакшене

Переменные окружения — стандартный способ конфигурировать контейнер. В продакшене `.env` на сервере не хранят — значения задают напрямую в системе деплоя (Railway, VPS, Kubernetes).

Для локальной разработки с docker-compose создай отдельный файл `.env` (он уже в `.dockerignore` и `.gitignore`):

```bash
# .env (не коммитить)
OPENWEATHER_API_KEY=your_key_here
JWT_SECRET=your_secret_here
PORT=3000
```

И `.env.example` (коммитить — показывает коллегам, какие переменные нужны):

```bash
# .env.example
OPENWEATHER_API_KEY=
JWT_SECRET=
PORT=3000
```

## 📖 Деплой на Railway

Railway — платформа, которая разворачивает приложение из GitHub-репозитория. Подходит для учебных проектов и MVP.

**Шаги:**

1. Зарегистрируйся на [railway.app](https://railway.app) и создай новый проект.
2. Нажми **Deploy from GitHub repo** и выбери репозиторий.
3. Railway автоматически обнаружит Dockerfile и соберёт образ.
4. Перейди в **Variables** и добавь переменные окружения:
   - `OPENWEATHER_API_KEY`
   - `JWT_SECRET`
   - `PORT` — Railway сам задаёт порт через `$PORT`, но можно указать явно.
5. Нажми **Deploy** — Railway соберёт образ и запустит контейнер.
6. Во вкладке **Settings → Networking** сгенерируй публичный домен.

Каждый push в `main` запускает новый деплой автоматически.

Для MongoDB — добавь новый сервис в проекте: **Add Service → Database → MongoDB**. Railway создаст контейнер и добавит переменную `MONGODB_URL` автоматически.

## 📖 Деплой на VPS

VPS (Virtual Private Server) — виртуальная машина в облаке с полным контролем. Подходит, когда нужна гибкость или Railway не хватает.

**Минимальная настройка сервера (Ubuntu 22.04):**

```bash
# Установить Docker
curl -fsSL https://get.docker.com | sh

# Добавить пользователя в группу docker (чтобы не писать sudo)
sudo usermod -aG docker $USER
newgrp docker
```

**Деплой через git:**

```bash
# На сервере: клонировать репозиторий
git clone https://github.com/your-username/weather-api.git
cd weather-api

# Создать .env с production-значениями
nano .env

# Поднять сервисы
docker compose up -d
```

**Обновление:**

```bash
git pull
docker compose build app
docker compose up -d app
```

Для production рекомендуют добавить Nginx как reverse proxy (обрабатывает HTTPS, отдаёт статику, проксирует запросы к Node.js) — это тема для отдельного раздела DevOps.

## 🎯 Финальное задание

Задеплой Weather API на Railway.

1. Убедись, что Dockerfile и `.dockerignore` в репозитории.
2. Убедись, что `.env` в `.gitignore` и не закоммичен.
3. Создай проект на Railway, подключи GitHub-репозиторий.
4. Добавь переменные окружения в Railway → Variables.
5. Добавь сервис MongoDB в Railway → Add Service.
6. Дождись деплоя и проверь публичный URL:

```bash
curl https://your-app.railway.app/api/weather/Лондон
```

Дополнительно: настрой автоматический деплой при пуше в `main` — Railway делает это по умолчанию, проверь в настройках.
