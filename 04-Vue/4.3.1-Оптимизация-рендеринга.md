# 4.3.1 Оптимизация рендеринга

Vue перерисовывает компоненты автоматически — это удобно. Но иногда он делает лишнюю работу: перерисовывает компоненты которые не изменились, или отслеживает объекты которые никогда не меняются. Этот раздел о том, как понять что именно происходит и как помочь Vue делать меньше работы.

---

## 🎯 Что ты узнаешь

- Как Vue решает что перерисовывать — виртуальный DOM и diffing
- `shallowRef` и `shallowReactive` — когда глубокое отслеживание лишнее
- `markRaw` — объекты которые никогда не должны быть реактивными
- `v-memo` — мемоизация части шаблона
- `defineAsyncComponent` — ленивая загрузка компонентов

---

## 📖 Как Vue решает что перерисовывать

Когда реактивные данные меняются, Vue не трогает реальный DOM сразу. Сначала он строит **виртуальный DOM** (vDOM) — легковесное JavaScript-представление того, как должен выглядеть интерфейс. Затем сравнивает новый vDOM со старым (**diffing**) и применяет к реальному DOM только минимально необходимые изменения.

```
Данные изменились
       ↓
Vue строит новый vDOM
       ↓
Сравнивает с предыдущим vDOM (diff)
       ↓
Находит различия
       ↓
Обновляет только изменившиеся узлы в реальном DOM
```

Перерисовка компонента — это пересчёт его шаблона в vDOM. Это быстро, но не бесплатно. Если компонент перерисовывается часто и шаблон сложный — это заметно.

### Что вызывает перерисовку

Компонент перерисовывается когда:
- Изменился реактивный `ref` или `reactive`, который читается в шаблоне или в `computed`
- Изменился prop, переданный в компонент
- Изменился слот

Всё остальное — нет. Переменная без `ref` не вызовет перерисовку при изменении.

## 📖 shallowRef и shallowReactive

`ref` по умолчанию делает объект **глубоко реактивным** — Vue рекурсивно оборачивает все вложенные свойства через Proxy. Для большого объекта с сотнями полей это лишняя работа если ты меняешь объект только целиком.

`shallowRef` отслеживает только замену значения целиком, не вложенные изменения:

```typescript
import { shallowRef } from "vue";

// ref — Vue следит за каждым полем вложенного объекта
const weather = ref({ temperature: 20, details: { humidity: 65, pressure: 1013 } });
weather.value.details.humidity = 70; // ✅ вызовет перерисовку

// shallowRef — Vue следит только за заменой .value
const weather = shallowRef({ temperature: 20, details: { humidity: 65, pressure: 1013 } });
weather.value.details.humidity = 70; // ❌ перерисовка не произойдёт
weather.value = { ...weather.value, temperature: 22 }; // ✅ вызовет перерисовку
```

`shallowReactive` — то же самое для `reactive`: отслеживает только верхний уровень:

```typescript
import { shallowReactive } from "vue";

const state = shallowReactive({
  city: "Москва",
  weather: { temperature: 20 }, // это поле не отслеживается глубоко
});

state.city = "Казань";           // ✅ перерисовка
state.weather.temperature = 22; // ❌ перерисовка не произойдёт
state.weather = { temperature: 22 }; // ✅ перерисовка — заменили само поле
```

**Когда использовать:** большие списки данных которые ты заменяешь целиком (результаты поиска, список погоды), объекты из API которые ты не мутируешь по частям.

---

## 📖 markRaw

Иногда в реактивном состоянии нужно хранить объект, который никогда не должен становиться реактивным — например, экземпляр класса, объект библиотеки или большой конфигурационный объект.

`markRaw` помечает объект так, что Vue никогда не будет оборачивать его в Proxy:

```typescript
import { ref, markRaw } from "vue";

// Без markRaw — Vue попытается сделать chart реактивным (это сломает Chart.js)
const chart = ref(new Chart(canvas, config)); // ❌

// С markRaw — Vue оставит объект как есть
const chart = ref(markRaw(new Chart(canvas, config))); // ✅

// Или: хранить в обычной переменной вне ref
let chart: Chart | null = null;
onMounted(() => {
  chart = new Chart(canvas, config);
});
```

Типичные кандидаты для `markRaw`: экземпляры Chart.js, Leaflet, Three.js, WebSocket, любые внешние классы с внутренним состоянием.

## 📖 v-memo

`v-memo` мемоизирует поддерево шаблона — Vue пропускает его обновление если перечисленные зависимости не изменились:

```vue
<template>
  <!-- Перерисовывает li только если изменился item.id или isSelected -->
  <li
    v-for="item in list"
    :key="item.id"
    v-memo="[item.id, isSelected(item.id)]"
  >
    <expensive-component :data="item" />
  </li>
</template>
```

Без `v-memo` каждый элемент списка перерисовывается при любом изменении родителя. С `v-memo` — только если изменилось что-то из зависимостей.

`v-memo` оправдан при рендеринге длинных списков (100+ элементов) с тяжёлыми дочерними компонентами. В обычных случаях он добавляет сложность без заметного эффекта.

## 📖 defineAsyncComponent

По умолчанию все компоненты входят в один JavaScript-бандл. `defineAsyncComponent` загружает компонент только когда он впервые нужен на странице:

```typescript
import { defineAsyncComponent } from "vue";

// Компонент загружается отдельным чанком только при первом рендеринге
const HeavyChart = defineAsyncComponent(
  () => import("./components/HeavyChart.vue")
);
```

С состояниями загрузки и ошибки:

```typescript
const HeavyChart = defineAsyncComponent({
  loader: () => import("./components/HeavyChart.vue"),
  loadingComponent: LoadingSpinner,   // пока грузится
  errorComponent: ErrorMessage,       // если не загрузился
  delay: 200,                         // задержка перед показом loadingComponent
  timeout: 5000,                      // через сколько мс считать ошибкой
});
```

Ленивая загрузка особенно полезна для тяжёлых компонентов (графики, редакторы, карты) и для всего что не нужно на первом экране.

## 📖 Когда вообще думать об оптимизации

Преждевременная оптимизация — источник большинства проблем. Правило простое:

1. Сначала пиши понятный код
2. Если что-то тормозит — **измерь** через Vue DevTools (вкладка Performance)
3. Только потом оптимизируй конкретное узкое место

Vue DevTools показывает сколько времени занял каждый рендер. Если компонент рендерится за 1-2мс — нет смысла его оптимизировать. Если за 50мс и это происходит часто — есть повод.

## 🔧 Практика 4.3.1.1: Ленивая загрузка компонентов

Сделаем `HeavyChart` (или любой тяжёлый компонент) асинхронным — он будет загружаться только когда пользователь открывает вкладку «Прогноз».

**Файл для изменения:** `src/views/WeatherView.vue`

**Стартовый код:**

```typescript
import { defineAsyncComponent } from "vue";

// TODO: замени статичный импорт ForecastList на defineAsyncComponent
// Было: import ForecastList from "../components/ForecastList.vue";
// Стало: const ForecastList = defineAsyncComponent(...)

// TODO: добавь loadingComponent: LoadingSpinner и delay: 200
```

**Ожидаемый результат:** открой Network в DevTools. При первом переключении на вкладку «Прогноз» появится отдельный сетевой запрос за чанком компонента. При повторном переключении — запроса нет, компонент уже в кэше.

<details>
<summary>💡 Подсказка</summary>

```typescript
import { defineAsyncComponent } from "vue";
import LoadingSpinner from "../components/LoadingSpinner.vue";

const ForecastList = defineAsyncComponent({
  loader: () => import("../components/ForecastList.vue"),
  loadingComponent: LoadingSpinner,
  delay: 200,
});
```
</details>

## 🔧 Практика 4.3.1.2: shallowRef для кэша погоды

В `weatherStore` кэш хранится как `ref<Map<string, WeatherData>>`. Vue отслеживает каждое поле каждого объекта `WeatherData` в Map — это лишнее, потому что мы никогда не мутируем объекты в кэше, только добавляем новые.

**Файл для изменения:** `src/stores/weather.ts`

**Стартовый код:**

```typescript
import { shallowRef, ref, computed } from "vue";

// TODO: замени ref на shallowRef для weatherCache
// Было: const weatherCache = ref<Map<string, WeatherData>>(new Map());
// Стало: const weatherCache = shallowRef(...)

// После записи в Map нужно явно сообщить Vue об изменении:
// weatherCache.value = new Map(weatherCache.value) — заменить на новый Map
// ИЛИ использовать triggerRef(weatherCache) после мутации
```

<details>
<summary>💡 Подсказка: два способа обновить shallowRef с Map</summary>

```typescript
import { shallowRef, triggerRef } from "vue";

const weatherCache = shallowRef<Map<string, WeatherData>>(new Map());

// Способ 1: заменить Map целиком (создаёт копию)
weatherCache.value.set(city, data);
weatherCache.value = new Map(weatherCache.value);

// Способ 2: мутировать и вызвать triggerRef вручную
weatherCache.value.set(city, data);
triggerRef(weatherCache); // явно говорим Vue "считай что значение изменилось"
```

`triggerRef` — более эффективен, не создаёт копию. Используй его когда производительность важна.
</details>

## 📝 Итоговое задание 4.3.1

**Задача:** профилировать приложение через Vue DevTools и найти компоненты с лишними перерисовками.

### Шаги:

1. Установи [Vue DevTools](https://devtools.vuejs.org/) для браузера если ещё не установлены

2. Открой приложение, перейди в DevTools → вкладка **Vue** → **Timeline**

3. Нажми «Record», переключи несколько городов, останови запись

4. Найди компоненты которые перерисовываются при каждом переключении города — проверь нужно ли это

5. Если `CityButton` перерисовывается для всех кнопок при выборе одной — добавь `v-memo`:

```vue
<!-- В HomeView.vue — v-for с CityButton -->
<CityButton
  v-for="city in filteredCities"
  :key="city"
  :city="city"
  :is-active="city === store.selectedCity"
  v-memo="[city === store.selectedCity]"
  @select="selectCity"
/>
```

6. Запиши ещё раз — убедись что теперь при выборе города перерисовываются только две кнопки: предыдущая активная и новая.
