# 4.3.3 SSR с Nuxt *(опционально)*

Vue SPA отлично работает для приложений, куда пользователь заходит авторизованным. Но если страницу должны находить поисковики или она должна открываться быстро на слабом интернете — нужен SSR. Nuxt берёт Vue 3 и добавляет серверный рендеринг, файловый роутинг и серверные API-маршруты.

---

## 🎯 Что ты узнаешь

- Чем Vue SPA отличается от SSR по существу
- Как устроен Nuxt-проект
- Файловый роутинг: страницы из файловой структуры
- `useFetch` и `useAsyncData` — загрузка данных на сервере
- SSR-безопасный код: что нельзя делать вне `onMounted`

## 📖 Зачем SSR

Vue SPA отдаёт браузеру пустой `index.html` и бандл JavaScript. Браузер скачивает JS, исполняет его, делает запрос к API, рендерит HTML. Пользователь видит контент только после всей этой цепочки.

При SSR сервер отдаёт уже готовый HTML с данными. Браузер показывает его сразу, потом «гидрирует» страницу — подключает Vue к уже готовому DOM.

Два практических следствия:

**SEO.** Поисковые боты видят готовый HTML без JavaScript. Для SPA они часто видят пустую страницу или не ждут выполнения скриптов.

**First Contentful Paint.** Пользователь видит контент раньше, чем скачается весь JavaScript.

Для погодного приложения с авторизацией SSR не нужен. Для публичной страницы прогноза, которую нужно индексировать по запросу «погода в Москве» — нужен.

## 📖 Создание проекта

```bash
npx nuxi@latest init weather-nuxt
cd weather-nuxt
npm install
npm run dev
```

Nuxt предложит выбрать пакетный менеджер и опциональные модули. Для старта достаточно базовой установки.

### Структура проекта

```
weather-nuxt/
├── pages/              ← файловый роутинг
│   ├── index.vue       ← маршрут /
│   └── weather/
│       └── [city].vue  ← маршрут /weather/:city
├── components/         ← автоимпорт компонентов
├── composables/        ← автоимпорт composables
├── server/
│   └── api/            ← серверные API-маршруты
├── nuxt.config.ts
└── .env
```

Nuxt автоматически импортирует компоненты из `components/` и composables из `composables/`. Импортировать вручную не нужно.

## 📖 Файловый роутинг

Роутер создаётся из структуры папки `pages/`. Не нужно писать `router.ts` и описывать маршруты вручную.

```
pages/
├── index.vue           → /
├── about.vue           → /about
└── weather/
    ├── index.vue       → /weather
    └── [city].vue      → /weather/Москва, /weather/Берлин
```

Динамический параметр в имени файла — `[city]`. Внутри компонента он доступен через `useRoute`:

```vue
<!-- pages/weather/[city].vue -->
<script setup lang="ts">
const route = useRoute();
const city = route.params.city as string;
</script>
```

`useRoute` в Nuxt — тот же `useRoute` что и во Vue Router. Импортировать не нужно — Nuxt добавит автоматически.

### Навигация

```vue
<!-- Декларативная через NuxtLink (аналог RouterLink) -->
<NuxtLink to="/weather/Москва">Погода в Москве</NuxtLink>

<!-- Программная -->
<script setup lang="ts">
const router = useRouter();
router.push(`/weather/${city}`);
</script>
```

## 📖 useFetch — загрузка данных на сервере

В SPA данные загружаются в `onMounted` — это клиент. При SSR запрос должен произойти на сервере, до отправки HTML браузеру. `useFetch` делает именно это.

```vue
<!-- pages/weather/[city].vue -->
<script setup lang="ts">
const route = useRoute();
const city = route.params.city as string;

const { data, status, error } = await useFetch(
  `https://api.openweathermap.org/data/2.5/weather`,
  {
    query: {
      q: city,
      appid: useRuntimeConfig().weatherApiKey,
      units: "metric",
      lang: "ru",
    },
  }
);
</script>

<template>
  <div v-if="status === 'pending'">Загрузка...</div>
  <div v-else-if="error">{{ error.message }}</div>
  <div v-else>{{ data?.name }}: {{ data?.main.temp }}°C</div>
</template>
```

`useFetch` возвращает реактивные `data`, `status` и `error`. На сервере запрос выполнится до рендеринга. На клиенте Nuxt использует кешированный результат — повторного запроса не будет.

### useRuntimeConfig для API-ключей

В SPA ключи хранятся в `VITE_WEATHER_API_KEY` и попадают в бандл. Это значит, что ключ виден в коде браузера.

В Nuxt серверный ключ хранится в `runtimeConfig` — он доступен только на сервере:

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  runtimeConfig: {
    weatherApiKey: process.env.WEATHER_API_KEY, // только сервер
    public: {
      appName: "Weather App", // сервер + клиент
    },
  },
});
```

```
# .env
WEATHER_API_KEY=your_key_here
```

Ключ из `runtimeConfig.weatherApiKey` не попадёт в JS-бандл, который скачивает браузер.

## 📖 useAsyncData — когда нужен контроль

`useFetch` удобен для прямых HTTP-запросов. Когда нужно вызвать функцию или сделать несколько запросов — используют `useAsyncData`:

```typescript
const { data, refresh } = await useAsyncData(
  `weather-${city}`,   // ключ кеша — должен быть уникальным
  async () => {
    const response = await fetch(
      `https://api.openweathermap.org/data/2.5/weather?q=${city}&appid=${apiKey}&units=metric&lang=ru`
    );
    if (!response.ok) throw new Error("Город не найден");
    return response.json();
  }
);
```

Ключ кеша важен. Два вызова с одинаковым ключом вернут один результат. Если ключ зависит от параметра (`weather-${city}`), при смене города будет новый запрос.

## 📖 useState — реактивное состояние между запросами

`ref` в Nuxt работает локально — каждый запрос получает свой экземпляр. Если нужно состояние, которое сохраняется между навигациями на клиенте, используют `useState`:

```typescript
// composables/useFavorites.ts
export const useFavorites = () => {
  const favorites = useState<string[]>("favorites", () => []);

  function toggle(city: string) {
    const idx = favorites.value.indexOf(city);
    if (idx === -1) {
      favorites.value.push(city);
    } else {
      favorites.value.splice(idx, 1);
    }
  }

  return { favorites, toggle };
};
```

`useState` с одним ключом (`"favorites"`) возвращает один и тот же `ref` везде, где вызывается. Это SSR-безопасная альтернатива Pinia для простых случаев.

## 📖 SSR-безопасный код

На сервере нет `window`, `document`, `localStorage`, `navigator`. Код, который обращается к ним, сломается при SSR.

Правило: всё, что зависит от браузерных API — только внутри `onMounted`:

```typescript
// ❌ Сломается на сервере
const width = window.innerWidth;

// ✅ Только на клиенте
onMounted(() => {
  const width = window.innerWidth;
});
```

Для `localStorage` Nuxt предоставляет `useCookie` — работает и на сервере, и на клиенте:

```typescript
const unit = useCookie<"metric" | "imperial">("unit", {
  default: () => "metric",
});
```

## 📖 useSeoMeta — метатеги для SEO

Одна из главных причин использовать SSR. Теги попадут в HTML до того, как поисковик увидит страницу:

```vue
<script setup lang="ts">
const route = useRoute();
const city = route.params.city as string;

const { data } = await useFetch(`https://api.openweathermap.org/data/2.5/weather`, {
  query: { q: city, appid: useRuntimeConfig().weatherApiKey, units: "metric", lang: "ru" },
});

useSeoMeta({
  title: () => `Погода в ${data.value?.name ?? city}`,
  description: () =>
    `Текущая погода в ${data.value?.name ?? city}: ${data.value?.main.temp}°C, ${data.value?.weather[0]?.description}`,
  ogTitle: () => `Погода в ${data.value?.name ?? city}`,
});
</script>
```
