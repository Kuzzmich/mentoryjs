# 5.2.4 Валидация (Zod)

Данным от клиента не всегда можно доверять. Пользователь может отправить строку вместо числа, пропустить обязательное поле или передать email без символа `@`. Без валидации эти данные доберутся до бизнес-логики и породят непредсказуемые ошибки. Zod — библиотека, которая описывает ожидаемую форму данных в виде схемы и проверяет входящие данные до того, как они попадут в контроллер.

---

## 🎯 Что ты узнаешь

- Как описывать схемы валидации через Zod
- Как выводить TypeScript-типы из схем через `z.infer<>`
- Как написать middleware-валидатор для тела запроса, параметров и query
- Как возвращать понятные ошибки валидации клиенту

## 📖 Установка и первая схема

```bash
npm install zod
```

Схема описывает ожидаемую структуру данных. Если данные не соответствуют схеме — Zod выбрасывает `ZodError` с подробным описанием всех нарушений:

```typescript
import { z } from "zod";

const AddCitySchema = z.object({
  city: z.string().min(1, "Название города не может быть пустым"),
});

// Проверка данных:
const result = AddCitySchema.safeParse({ city: "Москва" });

if (!result.success) {
  console.log(result.error.issues); // массив ошибок
} else {
  console.log(result.data.city); // "Москва" — TypeScript знает тип
}
```

`safeParse` — не бросает исключение, возвращает объект с `success: true/false`. Альтернатива — `parse`, которая бросает `ZodError` при ошибке.

## 📖 Типы из схем: z.infer

Главное преимущество Zod — типы не нужно писать отдельно. Тип выводится из схемы автоматически:

```typescript
const AddCitySchema = z.object({
  city: z.string().min(1),
  notes: z.string().optional(),
});

type AddCityBody = z.infer<typeof AddCitySchema>;
// Эквивалентно:
// type AddCityBody = {
//   city: string;
//   notes?: string | undefined;
// }
```

Схема — единственный источник правды: и для валидации, и для типов. Изменил схему — типы обновились автоматически.

## 📖 Основные методы схем

```typescript
// Строки
z.string()
z.string().min(2)                  // минимальная длина
z.string().max(100)                // максимальная длина
z.string().email()                 // формат email
z.string().url()                   // формат URL
z.string().regex(/^\d{4}$/)       // регулярное выражение

// Числа
z.number()
z.number().int()                   // целое число
z.number().positive()              // > 0
z.number().min(0).max(100)

// Булевые, перечисления
z.boolean()
z.enum(["metric", "imperial"])    // одно из значений
z.literal("admin")                // конкретное значение

// Объекты и массивы
z.object({ name: z.string() })
z.array(z.string())
z.array(z.string()).min(1)         // непустой массив

// Необязательные поля и значения по умолчанию
z.string().optional()              // string | undefined
z.string().nullable()              // string | null
z.string().default("metric")      // значение по умолчанию
```

## 📖 Middleware-валидатор

Валидацию удобно вынести в middleware — тогда контроллер получает уже проверенные данные и не знает о существовании Zod:

```typescript
// src/middleware/validate.ts
import { Request, Response, NextFunction } from "express";
import { ZodSchema, ZodError } from "zod";

type ValidateTarget = "body" | "params" | "query";

export function validate(schema: ZodSchema, target: ValidateTarget = "body") {
  return (req: Request, res: Response, next: NextFunction): void => {
    const result = schema.safeParse(req[target]);

    if (!result.success) {
      res.status(400).json({
        error: "Ошибка валидации",
        details: result.error.issues.map((issue) => ({
          field: issue.path.join("."),
          message: issue.message,
        })),
      });
      return;
    }

    // Заменяем исходные данные провалидированными (с дефолтами и преобразованиями)
    req[target] = result.data;
    next();
  };
}
```

Использование в роутере:

```typescript
import { validate } from "../middleware/validate";
import { z } from "zod";

const AddCitySchema = z.object({
  city: z.string().min(1, "Название города не может быть пустым"),
});

const WeatherQuerySchema = z.object({
  units: z.enum(["metric", "imperial"]).default("metric"),
});

router.post(
  "/",
  validate(AddCitySchema, "body"),
  asyncHandler(async (req, res) => {
    const { city } = req.body; // city гарантированно string
    const result = addFavorite(city);
    res.status(201).json(result);
  })
);

router.get(
  "/:city",
  validate(WeatherQuerySchema, "query"),
  asyncHandler(async (req, res) => {
    const units = req.query.units as "metric" | "imperial"; // уже провалидировано
    const weather = await fetchWeather(req.params.city, units);
    res.json(weather);
  })
);
```

## 📖 Валидация параметров маршрута

Параметры маршрута (`req.params`) — всегда строки. Но их тоже стоит валидировать: проверить формат, длину, допустимые символы:

```typescript
const CityParamSchema = z.object({
  city: z.string().min(1).max(100),
});

router.get(
  "/:city",
  validate(CityParamSchema, "params"),
  asyncHandler(async (req, res) => {
    const weather = await fetchWeather(req.params.city);
    res.json(weather);
  })
);
```

## 📖 Типобезопасность в контроллерах

Проблема: после `validate` middleware данные в `req.body` всё равно имеют тип `any`. TypeScript об этом не знает. Чтобы получить типобезопасный доступ, явно указывают тип:

```typescript
const RegisterSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
  name: z.string().min(2),
});

type RegisterBody = z.infer<typeof RegisterSchema>;

// В контроллере:
export async function register(
  req: Request<{}, {}, RegisterBody>,
  res: Response
): Promise<void> {
  const { email, password, name } = req.body;
  // TypeScript знает типы всех трёх полей
}
```

Схему и тип держат рядом с роутером или выносят в отдельный файл `schemas/`:

```
src/
├── schemas/
│   ├── auth.schemas.ts
│   ├── weather.schemas.ts
│   └── favorites.schemas.ts
```
