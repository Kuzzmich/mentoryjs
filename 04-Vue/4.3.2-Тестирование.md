# 4.3.2 Тестирование (Vitest + Vue Test Utils)

Компонент может выглядеть рабочим в браузере, но сломаться при следующем рефакторинге — и это никто не заметит до прода. Тесты фиксируют поведение: «кнопка Delete должна эмитить событие `remove` с именем города». Как только кто-то переименует событие или забудет передать аргумент — тест сломается сразу, а не в проде.

---

## 🎯 Что ты узнаешь

- Настройка Vitest в Vite-проекте
- Монтирование компонентов через Vue Test Utils
- Тестирование props, emit, пользовательских взаимодействий
- Мокирование fetch и composables
- Тестирование Pinia-сторов

---

## 📖 Vitest — тест-раннер для Vite

Vitest читает тот же `vite.config.ts`, что и приложение. Не нужны отдельный webpack, babel, jest-трансформеры. Тесты компилируются так же, как исходный код.

### Установка

```bash
npm install -D vitest @vue/test-utils jsdom
```

Добавь настройку в `vite.config.ts`:

```typescript
import { defineConfig } from "vite";
import vue from "@vitejs/plugin-vue";

export default defineConfig({
  plugins: [vue()],
  test: {
    environment: "jsdom",  // эмуляция браузерного DOM в Node.js
    globals: true,         // describe/it/expect без импортов
  },
});
```

Добавь в `tsconfig.json`:

```json
{
  "compilerOptions": {
    "types": ["vitest/globals"]
  }
}
```

Добавь скрипт в `package.json`:

```json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run"
  }
}
```

`vitest` запускает тесты в режиме watch; `vitest run` — однократно (для CI).

### Структура файлов

```
src/
├── components/
│   └── WeatherCard.vue
├── composables/
│   └── useWeather.ts
├── stores/
│   └── weather.ts
└── __tests__/          ← или рядом с файлом: WeatherCard.test.ts
    ├── WeatherCard.test.ts
    ├── useWeather.test.ts
    └── weatherStore.test.ts
```

Vitest найдёт любой файл с расширением `.test.ts`, `.spec.ts` или в папке `__tests__`.

## 📖 Основы Vue Test Utils

`mount` рендерит компонент в виртуальный DOM и возвращает `wrapper` — через него проверяют содержимое, кликают, находят элементы.

```typescript
import { mount } from "@vue/test-utils";
import WeatherCard from "../components/WeatherCard.vue";

describe("WeatherCard", () => {
  it("показывает название города из props", () => {
    const wrapper = mount(WeatherCard, {
      props: { city: "Москва" },
    });
    expect(wrapper.text()).toContain("Москва");
  });
});
```

### Основные методы wrapper

| Метод | Что делает |
|---|---|
| `wrapper.text()` | Весь текстовый контент |
| `wrapper.html()` | HTML-строка |
| `wrapper.find("selector")` | Первый элемент по CSS-селектору |
| `wrapper.findAll("selector")` | Все элементы |
| `wrapper.get("selector")` | Как `find`, но кидает если не найден |
| `wrapper.trigger("click")` | Эмулирует DOM-событие (Promise) |
| `wrapper.setValue("text")` | Устанавливает значение инпута |
| `wrapper.emitted("eventName")` | Массив всех эмитов события |
| `wrapper.props("propName")` | Текущее значение prop |
| `wrapper.classes()` | Массив CSS-классов корневого элемента |

### mount vs shallowMount

```typescript
// mount — рендерит полное дерево включая дочерние компоненты
const wrapper = mount(WeatherCard, { props: { city: "Москва" } });

// shallowMount — заменяет дочерние компоненты заглушками (stubs)
// Быстрее, изолирует тестируемый компонент
const wrapper = shallowMount(WeatherCard, { props: { city: "Москва" } });
```

Правило простое: `shallowMount` — когда тестируешь один компонент в изоляции и не хочешь, чтобы баг в дочернем компоненте ломал тест родителя.

## 📖 Тестирование пользовательских взаимодействий

```typescript
import { mount } from "@vue/test-utils";
import ErrorMessage from "../components/ErrorMessage.vue";

describe("ErrorMessage", () => {
  it("эмитит dismiss при клике на кнопку", async () => {
    const wrapper = mount(ErrorMessage, {
      props: { message: "Город не найден" },
    });

    // trigger возвращает Promise — нужен await
    await wrapper.find("button").trigger("click");

    // emitted() возвращает объект { eventName: [[...args], [...args]] }
    expect(wrapper.emitted("dismiss")).toBeTruthy();
    expect(wrapper.emitted("dismiss")?.[0]).toEqual([]);
  });

  it("отображает переданное сообщение", () => {
    const wrapper = mount(ErrorMessage, {
      props: { message: "Сеть недоступна" },
    });

    expect(wrapper.text()).toContain("Сеть недоступна");
  });
});
```

### Тестирование v-model и форм

```typescript
it("фильтрует города по вводу", async () => {
  const wrapper = mount(HomeView);
  const input = wrapper.find("input");

  await input.setValue("Мос");

  // После setValue Vue обновляет DOM — проверяем результат
  const cities = wrapper.findAll("[data-testid='city-item']");
  expect(cities).toHaveLength(1);
  expect(cities[0].text()).toContain("Москва");
});
```

> `data-testid` — атрибут без семантики, только для тестов. Лучше, чем привязываться к CSS-классам или тексту кнопок.

## 📖 Тестирование async-кода

Компонент запускает `fetch` в `onMounted`. К моменту первого рендера данных ещё нет — тест должен уметь ждать.

### flushPromises

```typescript
import { mount, flushPromises } from "@vue/test-utils";
import { vi } from "vitest";

// Мокируем глобальный fetch ДО монтирования компонента
beforeEach(() => {
  global.fetch = vi.fn().mockResolvedValue({
    ok: true,
    json: async () => ({
      name: "Moscow",
      sys: { country: "RU" },
      main: { temp: 5, feels_like: 2, humidity: 80 },
      weather: [{ description: "облачно", icon: "04d" }],
    }),
  });
});

afterEach(() => {
  vi.restoreAllMocks();
});

it("загружает и отображает данные о погоде", async () => {
  const wrapper = mount(WeatherCard, {
    props: { city: "Москва" },
  });

  // Ждём, пока все промисы завершатся (onMounted fetch)
  await flushPromises();

  expect(wrapper.text()).toContain("5");
  expect(wrapper.text()).toContain("облачно");
});
```

### Тестирование состояний загрузки и ошибки

```typescript
it("показывает спиннер во время загрузки", async () => {
  // fetch никогда не завершается в этом тесте
  global.fetch = vi.fn().mockReturnValue(new Promise(() => {}));

  const wrapper = mount(WeatherCard, { props: { city: "Москва" } });

  expect(wrapper.find("[data-testid='spinner']").exists()).toBe(true);
});

it("показывает ошибку если API вернул 404", async () => {
  global.fetch = vi.fn().mockResolvedValue({
    ok: false,
    status: 404,
  });

  const wrapper = mount(WeatherCard, { props: { city: "НесуществующийГород" } });
  await flushPromises();

  expect(wrapper.find("[data-testid='error']").exists()).toBe(true);
  expect(wrapper.text()).toContain("Город не найден");
});
```

## 📖 Тестирование composables

Composable без lifecycle-хуков можно звать прямо в тесте — Vue-контекст не нужен:

```typescript
import { useLocalStorage } from "../composables/useLocalStorage";

describe("useLocalStorage", () => {
  beforeEach(() => {
    localStorage.clear();
  });

  it("возвращает defaultValue если ключа нет", () => {
    const { value } = useLocalStorage("cities", ["Москва"]);
    expect(value.value).toEqual(["Москва"]);
  });

  it("персистирует значение в localStorage", () => {
    const { value } = useLocalStorage("cities", []);
    value.value = ["Москва", "Питер"];

    // Создаём новый экземпляр — должен прочитать из localStorage
    const { value: value2 } = useLocalStorage("cities", []);
    expect(value2.value).toEqual(["Москва", "Питер"]);
  });
});
```

Если composable использует `onMounted` или `onUnmounted`, его нужно вызывать в контексте компонента. Для этого пишут небольшую обёртку:

```typescript
import { createApp } from "vue";

function withSetup<T>(composable: () => T): T {
  let result!: T;
  const app = createApp({
    setup() {
      result = composable();
      return () => {};
    },
  });
  app.mount(document.createElement("div"));
  return result;
}

it("запрашивает данные при монтировании", async () => {
  global.fetch = vi.fn().mockResolvedValue({ ok: true, json: async () => mockData });

  const { weather, isLoading } = withSetup(() => useWeather("Москва"));

  expect(isLoading.value).toBe(true);
  await flushPromises();
  expect(isLoading.value).toBe(false);
  expect(weather.value).not.toBeNull();
});
```

## 📖 Тестирование Pinia-сторов

Стор тестируется отдельно, без компонентов. Каждый тест должен получить чистый экземпляр Pinia — иначе состояние из одного теста утечёт в следующий.

```bash
npm install -D @pinia/testing
```

```typescript
import { setActivePinia, createPinia } from "pinia";
import { useWeatherStore } from "../stores/weather";
import { vi } from "vitest";

describe("weatherStore", () => {
  beforeEach(() => {
    // Свежая Pinia для каждого теста
    setActivePinia(createPinia());
  });

  afterEach(() => {
    vi.restoreAllMocks();
  });

  it("fetchWeather сохраняет данные в кэш", async () => {
    global.fetch = vi.fn().mockResolvedValue({
      ok: true,
      json: async () => ({
        name: "Moscow",
        sys: { country: "RU" },
        main: { temp: 5, feels_like: 2, humidity: 80 },
        weather: [{ description: "облачно", icon: "04d" }],
      }),
    });

    const store = useWeatherStore();
    await store.fetchWeather("Москва");

    expect(store.weatherCache.has("Москва")).toBe(true);
    expect(store.weatherCache.get("Москва")?.temperature).toBe(5);
  });

  it("не делает повторный запрос если город уже в кэше", async () => {
    global.fetch = vi.fn().mockResolvedValue({
      ok: true,
      json: async () => mockApiResponse,
    });

    const store = useWeatherStore();
    await store.fetchWeather("Москва");
    await store.fetchWeather("Москва"); // второй вызов

    // fetch должен быть вызван только один раз
    expect(fetch).toHaveBeenCalledTimes(1);
  });

  it("устанавливает ошибку если API вернул не-ok ответ", async () => {
    global.fetch = vi.fn().mockResolvedValue({ ok: false });

    const store = useWeatherStore();
    await store.fetchWeather("НесуществующийГород");

    expect(store.error).not.toBeNull();
    expect(store.isLoading).toBe(false);
  });
});
```
