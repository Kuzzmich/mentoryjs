# 5.3.1 Тестирование API (Vitest + Supertest)

Ручная проверка через curl быстро перестаёт справляться: с каждым новым маршрутом нужно перепроверять всё остальное. Тесты решают это — запускаешь одну команду и видишь, что сломалось. Здесь разберём два уровня: unit-тесты для сервисов и интеграционные тесты для маршрутов через Supertest.

---

## 🎯 Что ты узнаешь

- Как настроить Vitest для Node.js-проекта
- Как тестировать сервисы через unit-тесты
- Как делать HTTP-запросы к Express в тестах через Supertest
- Как изолировать тесты: сбрасывать состояние и мокать модули

## 📖 Установка и настройка

```bash
npm install -D vitest supertest @types/supertest
```

Добавь конфигурацию в `package.json`:

```json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run"
  }
}
```

`vitest` — запускает тесты в watch-режиме. `vitest run` — один запуск, для CI.

Для работы с TypeScript и путями импортов создай `vitest.config.ts`:

```typescript
import { defineConfig } from "vitest/config";

export default defineConfig({
  test: {
    environment: "node",
    globals: true, // describe, it, expect без импорта
  },
});
```

## 📖 Структура тестов

Тесты кладут рядом с файлами, которые тестируют, с суффиксом `.test.ts`:

```
src/
├── services/
│   ├── favorites.service.ts
│   └── favorites.service.test.ts
├── routes/
│   ├── weather.router.ts
│   └── weather.router.test.ts
```

Базовая структура теста:

```typescript
import { describe, it, expect, beforeEach, vi } from "vitest";

describe("FavoritesService", () => {
  beforeEach(() => {
    // сброс состояния перед каждым тестом
  });

  it("добавляет город в список", () => {
    const result = addFavorite("Москва");
    expect(result).toContain("Москва");
  });

  it("не добавляет дубликат", () => {
    addFavorite("Москва");
    addFavorite("Москва");
    const result = getFavorites();
    expect(result).toHaveLength(1);
  });
});
```

## 📖 Unit-тесты для сервисов

Сервисы тестируются проще всего: они принимают данные и возвращают результат без HTTP. Главная проблема — состояние между тестами. Если один тест добавил город, следующий его унаследует.

Решение: экспортировать функцию сброса состояния или хранить данные в переменной, доступной для тестов:

```typescript
// src/services/favorites.service.ts
let favorites: string[] = [];

export function getFavorites(): string[] {
  return [...favorites]; // возвращаем копию
}

export function addFavorite(city: string): string[] {
  if (!favorites.includes(city)) {
    favorites.push(city);
  }
  return [...favorites];
}

export function removeFavorite(city: string): string[] | null {
  const index = favorites.indexOf(city);
  if (index === -1) return null;
  favorites.splice(index, 1);
  return [...favorites];
}

// Только для тестов
export function _resetFavorites(): void {
  favorites = [];
}
```

```typescript
// src/services/favorites.service.test.ts
import { describe, it, expect, beforeEach } from "vitest";
import {
  getFavorites,
  addFavorite,
  removeFavorite,
  _resetFavorites,
} from "./favorites.service";

describe("FavoritesService", () => {
  beforeEach(() => {
    _resetFavorites();
  });

  describe("addFavorite", () => {
    it("добавляет город в пустой список", () => {
      const result = addFavorite("Москва");
      expect(result).toEqual(["Москва"]);
    });

    it("не добавляет дубликат", () => {
      addFavorite("Москва");
      addFavorite("Москва");
      expect(getFavorites()).toHaveLength(1);
    });

    it("добавляет несколько разных городов", () => {
      addFavorite("Москва");
      addFavorite("Берлин");
      expect(getFavorites()).toEqual(["Москва", "Берлин"]);
    });
  });

  describe("removeFavorite", () => {
    it("удаляет существующий город", () => {
      addFavorite("Москва");
      const result = removeFavorite("Москва");
      expect(result).toEqual([]);
    });

    it("возвращает null если города нет", () => {
      const result = removeFavorite("Несуществующий");
      expect(result).toBeNull();
    });
  });
});
```

## 📖 Изоляция состояния между тестами

Это ключевая проблема при тестировании сервисов с внутренним состоянием. Если не сбрасывать данные между тестами, они «текут» из одного в другой.

**Что происходит без сброса:**

```typescript
describe("FavoritesService — ПЛОХО: без сброса", () => {
  it("добавляет город в пустой список", () => {
    const result = addFavorite("Москва");
    expect(result).toEqual(["Москва"]); // ✅ проходит
  });

  it("добавляет город в пустой список", () => {
    const result = addFavorite("Берлин");
    // ❌ падает: result = ["Москва", "Берлин"], а не ["Берлин"]
    // "Москва" осталась с предыдущего теста
    expect(result).toEqual(["Берлин"]);
  });
});
```

Тесты зависят от порядка запуска и мешают друг другу. Если запустить только второй тест — он пройдёт. Если оба вместе — упадёт. Такие тесты называют нестабильными (flaky).

**Решение: `_resetFavorites` + `beforeEach`**

В сервисе экспортируем функцию сброса — она нужна только для тестов:

```typescript
// src/services/favorites.service.ts
let favorites: string[] = [];

export function addFavorite(city: string): string[] {
  if (!favorites.includes(city)) {
    favorites.push(city);
  }
  return [...favorites];
}

// Только для тестов — сбрасывает массив в исходное состояние
export function _resetFavorites(): void {
  favorites = [];
}
```

В тестах `beforeEach` запускает `_resetFavorites()` **перед каждым** `it()`:

```typescript
describe("FavoritesService — ХОРОШО: со сбросом", () => {
  beforeEach(() => {
    _resetFavorites(); // ← вызывается перед каждым it()
  });

  it("добавляет город в пустой список", () => {
    // favorites = [] — гарантировано beforeEach
    const result = addFavorite("Москва");
    expect(result).toEqual(["Москва"]); // ✅
  });

  it("добавляет другой город в пустой список", () => {
    // favorites = [] — снова гарантировано beforeEach
    const result = addFavorite("Берлин");
    expect(result).toEqual(["Берлин"]); // ✅
  });
});
```

Порядок выполнения для двух тестов выглядит так:

```
beforeEach → it("добавляет Москву")
beforeEach → it("добавляет Берлин")
```

Каждый тест начинается с чистого состояния, независимо от того, что сделали предыдущие.

> **Соглашение:** функции только для тестов принято называть с подчёркивания — `_resetFavorites`. Это сигнал, что в production-коде их вызывать не нужно.

## 📖 Supertest — HTTP-тесты без запуска сервера

Supertest позволяет делать HTTP-запросы к Express-приложению без `app.listen`. Он сам поднимает временный сервер на произвольном порту и закрывает его после теста:

```typescript
import supertest from "supertest";
import app from "../index"; // экспортируй app без app.listen

const request = supertest(app);

it("возвращает список избранного", async () => {
  const response = await request.get("/api/favorites");
  expect(response.status).toBe(200);
  expect(response.body).toBeInstanceOf(Array);
});
```

Для этого нужно разделить создание `app` и запуск сервера:

```typescript
// src/app.ts — только создание приложения
import "dotenv/config";
import express from "express";
import helmet from "helmet";
import cors from "cors";
import weatherRouter from "./routes/weather.router";
import favoritesRouter from "./routes/favorites.router";
import authRouter from "./routes/auth.router";
import { errorHandler } from "./middleware/errorHandler";

export const app = express();

app.use(helmet());
app.use(cors());
app.use(express.json());

app.use("/api/weather", weatherRouter);
app.use("/api/favorites", favoritesRouter);
app.use("/api/auth", authRouter);

app.use(errorHandler);
```

```typescript
// src/index.ts — только запуск
import { app } from "./app";
import { config } from "./config";

app.listen(config.port, () => {
  console.log(`Сервер запущен: http://localhost:${config.port}`);
});
```

## 📖 Интеграционные тесты маршрутов

```typescript
// src/routes/favorites.router.test.ts
import { describe, it, expect, beforeEach } from "vitest";
import supertest from "supertest";
import { app } from "../app";
import { _resetFavorites } from "../services/favorites.service";

const request = supertest(app);

describe("GET /api/favorites", () => {
  beforeEach(() => {
    _resetFavorites();
  });

  it("возвращает пустой массив изначально", async () => {
    const res = await request.get("/api/favorites");
    expect(res.status).toBe(200);
    expect(res.body).toEqual([]);
  });
});

describe("POST /api/favorites", () => {
  beforeEach(() => {
    _resetFavorites();
  });

  it("добавляет город", async () => {
    const res = await request
      .post("/api/favorites")
      .send({ city: "Москва" });

    expect(res.status).toBe(201);
    expect(res.body).toContain("Москва");
  });

  it("возвращает 400 если city не передан", async () => {
    const res = await request
      .post("/api/favorites")
      .send({});

    expect(res.status).toBe(400);
    expect(res.body).toHaveProperty("error");
  });

  it("возвращает 400 если city пустая строка", async () => {
    const res = await request
      .post("/api/favorites")
      .send({ city: "" });

    expect(res.status).toBe(400);
  });
});
```

## 📖 Мокирование модулей

Когда сервис обращается к внешнему API (OpenWeatherMap), тест не должен делать реальный запрос — это медленно, ненадёжно и требует API-ключ. `vi.mock` заменяет модуль моком:

```typescript
// src/routes/weather.router.test.ts
import { describe, it, expect, vi, beforeEach } from "vitest";
import supertest from "supertest";
import { app } from "../app";

// Мокируем сервис целиком
vi.mock("../services/weather.service", () => ({
  fetchWeather: vi.fn(),
}));

import { fetchWeather } from "../services/weather.service";

const request = supertest(app);

describe("GET /api/weather/:city", () => {
  beforeEach(() => {
    vi.clearAllMocks();
  });

  it("возвращает данные о погоде", async () => {
    const mockWeather = {
      city: "Москва",
      temp: -5,
      feels_like: -9,
      description: "пасмурно",
      humidity: 80,
      wind_speed: 3.5,
    };

    vi.mocked(fetchWeather).mockResolvedValue(mockWeather);

    const res = await request.get("/api/weather/Москва");

    expect(res.status).toBe(200);
    expect(res.body).toEqual(mockWeather);
    expect(fetchWeather).toHaveBeenCalledWith("Москва", "metric");
  });

  it("возвращает 404 если город не найден", async () => {
    vi.mocked(fetchWeather).mockRejectedValue(
      new AppError("Город не найден", 404)
    );

    const res = await request.get("/api/weather/НесуществующийГород");

    expect(res.status).toBe(404);
    expect(res.body).toHaveProperty("error");
  });
});
```

## 📖 Тестирование авторизованных маршрутов

Маршруты с `authMiddleware` требуют токен. Его получают через регистрацию/вход прямо в тесте:

```typescript
describe("GET /api/favorites (авторизованный)", () => {
  let token: string;

  beforeEach(async () => {
    _resetFavorites();

    // Регистрируемся и получаем токен
    const res = await request
      .post("/api/auth/register")
      .send({ email: "test@mail.com", password: "password123", name: "Тест" });

    token = res.body.token;
  });

  it("возвращает список с токеном", async () => {
    const res = await request
      .get("/api/favorites")
      .set("Authorization", `Bearer ${token}`);

    expect(res.status).toBe(200);
  });

  it("возвращает 401 без токена", async () => {
    const res = await request.get("/api/favorites");
    expect(res.status).toBe(401);
  });
});
```

## 🎯 Финальное задание

Покрой тестами маршрут `GET /api/weather/:city` с мокированием `fetchWeather`:

1. Настрой `vi.mock` для `weather.service`
